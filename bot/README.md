# Бот выдачи ключей — @SnTAppsBot

Деньги принимает [lava.top](https://lava.top), бот только выдаёт ключ:

```
покупка на lava.top  →  вебхук в бота  →  заказ лёг в журнал
покупатель пишет боту почту  →  бот находит заказ  →  присылает ключ
```

Ни одного письма в схеме нет. SMTP на сервере закрыт, а квоту гугловского релея
занимает другой проект; телеграм же есть у всех, кто платит через lava.top.

Приложение проверяет ключ офлайн — сервер в проверке не участвует, бот только
подписывает и помнит, кому что выдал.

Один бот обслуживает несколько приложений: номер товара едет внутри ключа
(`SKU_*` в `license.py`), чужой ключ приложение не примет.

## Как работает ключ

```
FERN + base32( payload‖подпись )
       payload = версия(1) товар(1) номер(4) день выдачи(2)
       подпись = Ed25519 над payload
```

Приватный ключ живёт только у владельца: `~/.config/fern/license_ed25519.key`
(права 600). Публичная половина вшита в приложение —
`app/lib/services/license_service.dart`, константа `publicKeyBase64`.

Ключ **не привязан к устройству**: покупатель ставит приложение на все свои
телефоны. Утёкший ключ отзывается по номеру — номер вносится в `revoked` в том
же файле и уезжает с ближайшим обновлением.

## Файлы

| Файл | Что делает |
|---|---|
| `bot.py` | телеграм-бот и приёмник вебхука |
| `lava.py` | разбор payload от lava.top |
| `store.py` | журнал заказов и выдачи |
| `texts.py` | сообщения покупателю и владельцу |
| `license.py` | подпись ключей, `--id N` для ручной выдачи |
| `test_*.py` | проверки, `python3 bot/test_lava.py` и так далее |

Разбор вебхука, журнал и тексты вынесены отдельно намеренно: ошибка в первых
двух означает либо невыданный ключ честному покупателю, либо выданный чужой, а
незакрытая угловая скобка в почте роняет отправку сообщения целиком. Такое
проверяют тестом, а не глазами — и без aiogram, которого на машине
разработчика может не быть:

```bash
for t in bot/test_*.py; do python3 "$t"; done   # 102 проверки, сеть не нужна
```

`test_bot.py` проходит путь целиком — вебхук, журнал, ответ покупателю,
возврат — подменяя aiogram заглушками.

## Команды

| Команда | Кому | Что делает |
|---|---|---|
| `/start` | всем | витрина магазина с кнопками |
| `/buy` | всем | карточка Fern Pro и ссылка на оплату |
| `/key` | всем | показать свои ключи снова |
| `/help` | всем | как получить и активировать |
| `/stats` | владельцу | продажи, выдачи, возвраты |
| `/find почта` | владельцу | что числится за почтой, ничего не меняет |
| `/grant почта` | владельцу | выдать ключ вручную |

Владельческие три видны только в чате владельца (`SNT_BOT_OWNER`).

`/grant` не проставляет получателя: покупатель придёт к боту сам и заберёт тот
же ключ, а не второй.

Почта — единственное доказательство покупки, поэтому перебор чужих адресов
останавливается после пяти промахов за час (`MISS_LIMIT` в `store.py`).

## Запуск

```bash
python3 -m venv venv && ./venv/bin/pip install -r requirements.txt

export SNT_BOT_TOKEN=...        # от @BotFather
export SNT_BOT_OWNER=...        # ваш telegram id, для /stats
export SNT_WEBHOOK_KEY=...      # тот же секрет, что вписан в lava.top
./venv/bin/python bot.py
```

| Переменная | Зачем | По умолчанию |
|---|---|---|
| `SNT_BOT_TOKEN` | токен бота | обязательна |
| `SNT_BOT_OWNER` | telegram id владельца | 0 |
| `SNT_WEBHOOK_KEY` | секрет заголовка `X-Api-Key` | пусто (вебхук открыт — так не оставлять) |
| `SNT_WEBHOOK_PORT` | порт приёмника | 8091 |
| `SNT_BOT_DB` | файл журнала | `purchases.db` рядом |
| `SNT_BOT_PRODUCTS` | JSON: товар lava.top → sku и название | Fern Pro |
| `SNT_BUY_URL` | ссылка кнопки «Купить» | страница Fern Pro |
| `FERN_LICENSE_KEY_PATH` | приватный ключ подписи | `~/.config/fern/license_ed25519.key` |

## Вебхук

Бот слушает `127.0.0.1:8091/lava` — наружу его выставляет обратный прокси,
lava.top ходит только по HTTPS. Пример для Caddy:

```
store.example.com {
    reverse_proxy /lava 127.0.0.1:8091
}
```

В личном кабинете lava.top: адрес `https://store.example.com/lava`,
аутентификация по API-ключу, значение — то же, что в `SNT_WEBHOOK_KEY`.

Проверить, что живой: `curl https://store.example.com/health` → `alive`.

**Имена полей в payload взяты с запасом.** Документация lava.top описывает их
словами, а не схемой, поэтому `lava.py` ищет значения по всей структуре, а бот
пишет сырой payload в лог и в колонку `raw`. После первой продажи посмотрите
лог и, если формат оказался другим, сузьте списки ключей в `lava.py`.

## Добавить второе приложение

1. Завести товар на lava.top, взять его идентификатор из ссылки.
2. Добавить номер в `SKU_*` (`license.py`) и то же значение в приложение.
3. Дописать товар в `SNT_BOT_PRODUCTS`:

```json
{
  "34586da0-...": {"sku": 1, "name": "Fern Pro",  "app": "Fern"},
  "…":            {"sku": 2, "name": "Kadr Pro",  "app": "Kadr"}
}
```

## Ручная выдача

Пока вебхук не настроен или заказ потерялся:

```bash
python3 bot/license.py --id 42            # ключ №42
python3 bot/license.py --verify FERN...   # проверить чужой ключ
python3 bot/license.py --pubkey           # публичный ключ для приложения
```

Номер берите следующий за максимальным в журнале, иначе два человека получат
один ключ и отзыв заденет обоих.

## Возврат

lava.top возвращает деньги сама и присылает событие тем же вебхуком. Бот
помечает заказ возвращённым (ключ по нему больше не выдаётся и пропадает из
«Моих ключей») и пишет владельцу номер выданной лицензии.

Дальше руками: внести номер в `revoked` в `license_service.dart` и выпустить
обновление. Проверка ключа идёт офлайн, поэтому уже выданный ключ иначе
продолжит работать.
